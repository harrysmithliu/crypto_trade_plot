## 1. 文档信息

| 项目 | 内容 |
|---|---|
| 项目名称 | Crypto Trade Daily Report |
| 当前版本 | v0.1（演示版） |
| 文档状态 | 待评审 |

## 2. 项目背景

运营团队需要每天查看加密货币支付业务的交易规模、成功率、币种偏好和处理延迟。目前缺少一个可快速验证指标口径与展示效果的最小分析闭环。

本项目第一版以近 30 天模拟交易数据为输入，使用 Pandas 完成清洗、聚合和基础异常检测，使用 Plotly 生成可独立打开的交互式 HTML 报告。第一版以演示和指标验证为目标，同时通过清晰的模块边界、配置管理和数据接口，为后续接入 PostgreSQL、自动调度、消息通知及运行监控保留扩展能力。

## 3. 项目目标

### 3.1 本版目标

1. 建立从 CSV 数据输入到 HTML 报告输出的完整可运行闭环。
2. 统一核心运营指标的计算口径。
3. 提供三类可交互图表，支持快速识别交易趋势、币种结构和延迟问题。
4. 对异常失败率进行规则化标记，并在报告中显著展示。
5. 代码结构可在不重写分析和可视化逻辑的前提下切换到数据库数据源。
6. 提供清晰的运行说明、配置示例和最小自动化测试。

### 3.2 成功标准

- 新环境按照 README 操作可以生成报告。
- 同一份输入数据重复执行，聚合结果保持一致。
- 报告无需启动 Web 服务即可在现代浏览器中打开并交互。
- 三张核心图表及摘要指标均满足本文件的指标口径与验收条件。
- 数据异常、输入缺失或输出失败时，程序返回非零退出码并给出可定位的日志。
- 数据读取层可通过统一接口替换为 PostgreSQL 实现，业务聚合代码无需修改。

## 4. 范围定义

### 4.1 本版包含

- 生成或提供近 30 天 Mock CSV 交易数据。
- CSV 读取、字段校验、清洗、时区转换及去重。
- 按日聚合交易指标。
- 基于规则的每日失败率异常检测。
- 摘要指标卡片。
- 双轴趋势图、币种环形图、金额与延迟散点图。
- 生成单个独立 HTML 报告。
- 命令行执行入口、配置文件、日志和基础测试。
- 为数据库、调度、Email/Slack 通知预留接口。

### 4.2 本版不包含

- 实际接入生产 PostgreSQL。
- 实际部署 Cron、Airflow 或云端调度任务。
- 实际发送 Email 或 Slack 消息。
- 用户登录、权限控制或 Web Dashboard。
- 实时或分钟级数据刷新。
- 机器学习异常检测。
- 多币种汇率换算；输入中的 `fiat_volume_cad` 被视为已完成折算。
- 商户级下钻页面、错误码根因分析和告警处置工作流。

以上能力不在演示版验收范围内，但设计时不得形成明显的迁移阻碍。

## 5. 用户与使用场景

### 5.1 目标用户

- 运营人员：查看每日流水、成功率、币种偏好和异常日期。
- 技术/支持人员：查看延迟分布和异常订单线索。
- 数据分析人员：验证数据口径，并为后续分析扩展提供基础。

### 5.2 核心使用流程

1. 用户准备符合数据契约的 CSV 文件，或运行 Mock 数据生成脚本。
2. 用户通过命令行执行日报生成程序。
3. 程序读取配置与数据，执行校验、清洗、聚合和异常检测。
4. 程序在指定目录生成带日期版本的 HTML 报告及 `latest` 报告。
5. 用户在浏览器中打开报告，使用悬停、缩放和图例筛选进行分析。
6. 若执行失败，用户通过控制台输出和日志文件定位问题。

## 6. 数据需求

### 6.1 输入数据契约

CSV 文件使用 UTF-8 编码，首行为字段名。必需字段如下：

| 字段 | 类型 | 必填 | 规则 | 示例 |
|---|---|---:|---|---|
| `tx_id` | string | 是 | 交易唯一标识；非空 | `TX202606290001` |
| `timestamp` | datetime string | 是 | ISO 8601 UTC 时间；必须可解析 | `2026-06-29T12:30:00Z` |
| `merchant_id` | string | 是 | 非空 | `M1001` |
| `crypto_asset` | category | 是 | `USDC`、`USDT`、`ETH`、`BTC` | `USDC` |
| `fiat_volume_cad` | decimal | 是 | 大于或等于 0 | `1250.50` |
| `tx_status` | category | 是 | `Completed`、`Pending`、`Failed` | `Completed` |
| `latency_ms` | integer | 是 | 大于或等于 0 | `842` |

### 6.2 Mock 数据要求

- 覆盖连续 30 个自然日，每日均有交易。
- 默认至少生成 3,000 条记录，数量可通过配置调整。
- 正常日期失败率约为 2%，至少注入 1 个失败率约为 15% 的异常日期。
- 四种币种均应出现，默认让 USDC 占比最高。
- 金额与延迟应具有合理长尾，并包含少量大额、高延迟订单。
- 使用固定随机种子时，生成结果必须可复现。
- Mock 数据不得包含真实客户、商户或交易信息。

### 6.3 数据校验与清洗规则

1. 校验必需字段是否齐全；缺列时终止执行。
2. 将 `timestamp` 解析为 UTC，再转换为 `America/Toronto`，不得使用固定 `EST` 偏移，以正确处理夏令时。
3. 对 `tx_id` 去重；同一文件中重复记录默认保留最后一条，并记录重复数量。
4. 将金额与延迟转换为数值类型。
5. 无法解析的时间、非法状态、非法币种、负金额或负延迟均视为无效记录。
6. 默认删除无效记录并记录原因与数量；当无效记录比例超过配置阈值时终止报告生成。
7. 分析窗口以本地日期为准，默认取数据中最新日期及其之前共 30 个自然日。
8. 清洗完成后若无有效记录，程序必须终止执行。

## 7. 指标口径

所有日报指标以转换后的 `America/Toronto` 本地自然日统计。

| 指标 | 口径 |
|---|---|
| 每日提交订单数 | 当日全部状态的交易笔数 |
| 每日终态订单数 | 当日 `Completed + Failed` 的交易笔数 |
| 每日完成订单数 | 当日 `Completed` 交易笔数 |
| 每日失败订单数 | 当日 `Failed` 交易笔数 |
| 每日 Pending 数 | 当日 `Pending` 交易笔数 |
| 每日结算总额 | 当日 `Completed` 订单的 `fiat_volume_cad` 之和 |
| 每日成功率 | `Completed / (Completed + Failed) × 100%` |
| 每日失败率 | `Failed / (Completed + Failed) × 100%` |
| 平均延迟 | `Completed` 订单 `latency_ms` 的算术平均值 |
| P95 延迟 | `Completed` 订单 `latency_ms` 的第 95 百分位数 |
| 币种笔数占比 | 各币种 `Completed` 订单笔数 / 全部 `Completed` 订单笔数 |

当某日不存在终态订单时，成功率与失败率显示为空，不得显示为 0%。金额统一以 CAD 展示，默认保留两位小数；百分比默认保留两位小数。

## 8. 异常检测需求

### 8.1 默认规则

某日同时满足以下条件时标记为失败率异常：

1. 当日终态订单数大于或等于 `min_terminal_tx`，默认值为 50；
2. 当日失败率大于或等于 `absolute_failure_rate_threshold`，默认值为 10%；
3. 当日失败率大于或等于之前 7 个有效自然日平均失败率的 `relative_failure_multiplier` 倍，默认值为 3。

若之前不足 3 个有效基准日，只使用条件 1 和条件 2，并在结果中标记基准数据不足。

### 8.2 异常输出

聚合结果至少包含：

- `is_anomaly`
- `anomaly_reason`
- `baseline_failure_rate`
- `failure_rate`
- `terminal_tx_count`

报告摘要区显示异常天数；趋势图对异常日期使用红色标记，悬停时显示异常原因。

## 9. 报告需求

### 9.1 报告总体结构

报告按以下顺序展示：

1. 报告标题、生成时间和数据统计窗口。
2. 数据质量摘要：输入行数、有效行数、无效行数、重复行数。
3. 关键指标卡片。
4. 日常流水与成功率趋势图。
5. 支付币种偏好环形图。
6. 大额订单与处理延迟散点图。
7. 异常日期明细表。
8. 指标口径与报告备注。

### 9.2 摘要指标卡片

至少包含：

- 30 天累计结算金额。
- 最新完整日期的结算金额。
- 最新完整日期的成功率。
- 30 天累计 Pending 数量。
- 异常日期数量。
- 30 天 Completed 订单 P95 延迟。

### 9.3 图表一：日常流水与成功率趋势

- X 轴：本地日期。
- 左 Y 轴：每日结算总额，使用柱状图，单位 CAD。
- 右 Y 轴：每日成功率，使用折线图，显示为百分比。
- 悬停信息：日期、结算金额、成功率、完成数、失败数、Pending 数和异常状态。
- 异常日期使用红色点或其他醒目标记。
- 支持缩放、平移、图例开关和图片导出等 Plotly 标准交互。

### 9.4 图表二：支付币种偏好

- 图表类型：环形图。
- 数据范围：统计窗口内所有 `Completed` 订单。
- 默认口径：交易笔数占比。
- 展示内容：币种、交易笔数、占比和对应结算金额。
- 点击图例可隐藏或恢复币种，图表对当前可见类别重新显示占比。
- 无 Completed 订单时显示明确的空状态说明，不得生成误导性饼图。

### 9.5 图表三：大额订单与延迟监控

- 数据范围：统计窗口内所有 `Completed` 订单。
- X 轴：`fiat_volume_cad`；默认使用对数轴，以适应长尾金额。
- Y 轴：`latency_ms`。
- 颜色：`crypto_asset`。
- 悬停信息：交易号、商户、币种、日期、金额和延迟。
- 增加整体 P95 延迟参考线。
- 数据点过多时允许通过配置抽样，但大额订单和高延迟订单必须保留；默认 Mock 数据规模无需抽样。

### 9.6 HTML 输出

- 默认文件名：`Crypto_Trade_Daily_Report_YYYY-MM-DD.html`。
- 同时生成或覆盖 `Crypto_Trade_Daily_Report_latest.html`。
- HTML 必须内嵌 Plotly JavaScript，可在离线环境独立打开。
- 页面在常见桌面分辨率下布局完整，无横向内容溢出。
- 报告标题和正文均使用 UTF-8，中文不得乱码。

## 10. 命令行与配置需求

### 10.1 命令行入口

建议入口：

```bash
python run_daily.py --config config.yaml
```

至少支持以下参数或等价配置：

- 输入 CSV 路径。
- 输出目录。
- 统计窗口天数。
- 本地时区。
- 异常检测阈值。
- 日志级别。
- 是否生成 Mock 数据。
- 随机种子与 Mock 数据规模。

命令执行成功返回退出码 0；输入、处理或输出失败返回非零退出码。

### 10.2 配置原则

- 业务阈值、文件路径和环境相关参数不得散落在代码中。
- 配置提供合理默认值，并允许命令行覆盖。
- 演示版不得在配置或代码中保存数据库密码、Webhook 或其他密钥。
- 未来敏感参数通过环境变量或密钥管理服务注入。

### 11.2 数据源抽象

数据读取层应提供统一接口，例如：

```python
class TransactionDataSource:
    def load(self, start_at, end_at):
        """返回符合输入数据契约的 DataFrame。"""
```

本版实现 `CsvTransactionDataSource`。后续增加 `PostgresTransactionDataSource` 时，清洗、聚合、异常检测、图表和报告模块不得依赖 SQL 或数据库连接细节。

### 11.3 调度与通知扩展

- `run_daily.py` 必须可无交互执行，以便后续由 Cron、Airflow 或云端调度器调用。
- 主流程应返回结构化运行结果，包括报告路径、数据窗口、异常数量和执行状态。
- 后续通知模块只消费运行结果与报告路径，不重新计算业务指标。
- 本版可以提供通知接口占位或设计说明，但不得发起真实外部请求。

### 11.4 处理流程

```text
配置加载
  → 数据源读取
  → 数据契约校验
  → 数据清洗
  → 指标聚合
  → 异常检测
  → 图表生成
  → HTML 组装与写入
  → 运行结果与日志输出
```

## 12. 非功能需求

### 12.1 性能

- 在普通开发电脑上处理 100,000 条交易并生成报告，目标耗时不超过 30 秒。
- 不要求针对百万级以上数据优化；后续数据库版应在 SQL 层限制时间窗口和字段。

### 12.2 可维护性

- 核心清洗、指标和异常逻辑必须为独立函数或模块。
- 关键函数包含类型标注和简短说明。
- 不得在图表函数中重复计算核心业务指标。
- 代码风格保持一致，建议使用 Ruff 或等价工具检查。

### 12.3 可观测性

每次运行至少记录：

- 运行开始和结束时间。
- 输入文件与输出文件路径。
- 数据统计窗口。
- 输入、有效、无效及重复记录数。
- 异常日期数量。
- 各处理阶段耗时。
- 错误类型与可定位的错误信息。

日志不得输出未来可能出现的数据库密码、Webhook 或完整敏感交易载荷。

### 12.4 可重复性

- 依赖项必须记录在项目依赖文件中。
- Mock 数据生成支持固定随机种子。
- 同一配置和同一输入必须生成相同指标结果。
- 报告中的生成时间可以不同，但业务指标不得漂移。

## 13. 测试要求

### 13.1 单元测试

至少覆盖：

- UTC 到 `America/Toronto` 的日期转换，包括夏令时日期。
- 重复交易处理。
- 非法数据过滤和质量阈值。
- 每日结算金额计算。
- 成功率、失败率及零终态订单处理。
- 异常规则的正常、异常和低样本场景。
- 币种占比汇总。

### 13.2 集成测试

使用固定的小型 CSV 测试数据运行完整流程，验证：

- 命令返回成功状态。
- 带日期报告和 `latest` 报告均被生成。
- HTML 包含三张图表标题、摘要指标和异常明细。
- 对错误输入执行时不生成伪成功报告。

### 13.3 人工验收

- 在 Chrome、Edge 或 Safari 任一当前主流版本中打开报告。
- 验证悬停提示、缩放、图例隐藏与恢复正常。
- 验证异常日期在趋势图和明细表中一致。
- 断开网络后重新打开报告，图表仍可使用。
- 验证中文、金额和百分比格式正确。

## 14. 验收标准

满足以下全部条件视为演示版完成：

1. 项目能够生成符合要求的 30 天 Mock CSV。
2. 可通过单条命令完成从数据读取到 HTML 输出。
3. 报告包含规定的摘要指标、三张图和异常明细表。
4. 指标计算与第 7 节口径一致。
5. Mock 数据中注入的高失败率日期被正确标记。
6. 报告可离线打开，交互功能正常。
7. 非法输入能够产生清晰错误日志和非零退出码。
8. 关键业务逻辑具备自动化测试且测试通过。
9. README 包含安装、生成 Mock 数据、生成报告和测试命令。
10. 数据源读取与分析模块解耦，代码评审确认后续可增加 PostgreSQL 实现。

## 17. 后续稳定运行版演进路线

演示版验收后，可按以下顺序演进：

1. 实现 PostgreSQL 数据源，并增加连接重试、超时和凭证管理。
2. 将本地日志升级为结构化日志，并接入运行监控与失败告警。
3. 通过 Cron、Airflow 或云端调度器在多伦多时间每天 08:00 执行。
4. 将报告上传到内部对象存储，并通过 Email 或 Slack 分发链接。
5. 增加任务幂等、通知重试、重复发送保护和报告留存策略。
6. 增加商户、币种、网络和错误码维度的异常下钻。
7. 根据历史数据量升级异常算法，并引入数据质量趋势监控。

稳定运行版需要单独评审安全、权限、数据留存、基础设施和运维责任，不属于本版隐含交付范围。
